# ЛовиМить — адмінка: технічна специфікація

React-застосунок → Vercel, той самий backend/Supabase, що й основний Mini App.
Окрема авторизація для адміністраторів (не через Telegram, а email/пароль
або magic link через Supabase Auth).

## 1. Структура репозиторію

## 2. Авторизація адміністраторів

- Окрема таблиця `admin_users` (id, email, role, created_at) — role: super_admin / moderator
- Вхід через Supabase Auth (email + пароль)
- RLS-політики: доступ до "адмінських" операцій (видалення, зміна ролей) тільки
  для записів у admin_users з роллю super_admin
- Захищені роути на фронтенді (redirect на /login, якщо не авторизований)

## 3. Розділ "Категорії"

CRUD для таблиці `categories`.

Поля форми:
- назва (мультимовна — див. розділ "Переклади" нижче, тут просто ключ перекладу)
- іконка (вибір із бібліотеки іконок або завантаження SVG)
- порядок сортування (drag-and-drop список або поле number)
- активна/неактивна (щоб приховати категорію, не видаляючи)

Дії: додати, редагувати, видалити (з підтвердженням, якщо є пов'язані events),
змінити порядок перетягуванням.

## 4. Розділ "Переклади"

Окрема таблиця `translations`:

| поле | тип | опис |
|---|---|---|
| id | uuid | PK |
| key | text | унікальний ключ (напр. `category.sport`, `button.create_event`) |
| lang | text | код мови (uk, ru, en...) |
| value | text | сам переклад |
| updated_at | timestamptz | |

Унікальний індекс на `(key, lang)`.

Інтерфейс:
- таблиця: рядки = ключі, колонки = мови (uk/ru/en і т.д.), редагування inline
  прямо в таблиці (клік по клітинці → textarea → збереження)
- кнопка "додати нову мову" — додає нову колонку, генерує порожні значення
  для всіх існуючих ключів цієї мови
- кнопка "додати новий ключ" — для нових текстів у застосунку
- фільтр/пошук по ключу
- індикатор "немає перекладу" (порожня клітинка підсвічується) — щоб бачити
  прогалини одразу
- опціонально: імпорт/експорт у JSON (щоб фронтенд міг просто підвантажувати
  готовий файл локалізації, а не робити запит на кожен ключ)

Frontend Mini App при старті підвантажує всі переклади поточної мови користувача
одним запитом і кешує в localStorage/пам'яті сесії.

## 5. Розділ "Статуси-бейджі"

CRUD для таблиці `status_tags_catalog` — статуси, які учасники ставлять одне
одному після заходу (напр. "Гуру шашлику", "Жмот", "Душа компанії").

Поля форми:
- ключ (унікальний, для прив'язки до перекладів — `tag.<key>`)
- назва — мультимовна, редагується через розділ "Переклади" за ключем `tag.<key>`
- іконка/емодзі
- sentiment: positive / playful_negative (перемикач, впливає на візуальне
  групування на екрані оцінки після заходу)
- активний/неактивний

Окремо на цій сторінці — **статистика зловживань**: список тегів
playful_negative з найбільшою кількістю скарг або з підозрілою концентрацією
(один і той самий from_user_id ставить один тег різним людям занадто часто) —
щоб можна було вручну деактивувати проблемний тег або розібратися з
користувачем через розділ "Скарги".

## 6. Розділ "Користувачі"

Таблиця з пошуком/фільтрами по `users`:
- пошук за іменем/telegram_id
- фільтр за is_verified, is_pro
- сортування за rating_avg, created_at
- перегляд картки користувача: статистика, історія заходів, відгуки
- дії модератора: заблокувати/розблокувати, зняти верифікацію, вручну
  підтвердити верифікацію

## 7. Розділ "Заходи"

Таблиця `events` з фільтрами за статусом (planned/gathering/active/completed/
cancelled), категорією, датою.

- перегляд деталей заходу (учасники, чат — можна тільки переглядати,
  без редагування повідомлень)
- можливість примусово скасувати захід (наприклад, за скаргою)
- статистика: скільки створено за період, середня заповнюваність

## 8. Розділ "Скарги" (Reports)

Черга скарг зі статусами: нова / на розгляді / вирішена / відхилена.

- перегляд контексту скарги (захід, обидва учасники, причина, коментар)
- дії: попередити користувача, заблокувати, відхилити скаргу
- лог дій модератора (хто, коли, яке рішення) — окрема таблиця `moderation_log`

## 9. Розділ "Дашборд" (головна сторінка)

Базові метрики за період (день/тиждень/місяць):
- нових користувачів
- створених заходів
- активних заходів зараз
- зірок витрачено (симпатії + подарунки + PRO)
- відкритих скарг

## 10. Технічні примітки

- Supabase Row Level Security має бути налаштований так, щоб admin_users міг
  читати/писати в усі таблиці, а звичайний користувач Mini App — лише свої дані
- Для операцій, що потребують service_role (bypass RLS) — робити через backend
  API-ендпоінти (`/admin/...`), а не напряму з admin-фронтенду, щоб
  service_role ключ ніколи не потрапив у клієнтський код
- Роут `/admin/*` на backend захищений перевіркою токена Supabase Auth сесії
  адміністратора
